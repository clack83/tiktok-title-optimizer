## Context

基于已有的分类提示词系统和 DeepSeek 优化引擎，新增爆款标题种子库。由于个人开发者无法接入抖音 API 获取真实数据，采用 DeepSeek 模型自动生成各垂直领域的参考标题，作为优化时的 few-shot 示例和用户的灵感来源。

## Goals / Non-Goals

**Goals:**
- 每个分类预置 15-20 条高质量种子标题（DeepSeek 生成 + 人工筛选）
- 优化时将同分类 3-5 条种子标题注入 prompt 作为 few-shot 示例
- 用户在 Optimizer 页面可浏览当前分类的种子标题，获取灵感
- 支持通过 Celery 定时任务每周自动刷新种子库

**Non-Goals:**
- 不接入抖音 API 获取真实数据
- 不支持用户手动添加种子标题（V1 全是模型生成）
- 不做种子标题的真实效果追踪（无法获取真实播放量）

## Decisions

### 1. 种子标题生成: DeepSeek 批量生成 + 评分过滤
- **选择**: 使用 DeepSeek API 为每个分类生成候选标题，用标题评分引擎打分，只保留 >= 70 分的标题
- **理由**: 模型生成可以保证数量，评分引擎过滤可以保证质量。避免低质量种子标题影响优化效果。

```
生成流程:
┌─────────────┐    ┌──────────────┐    ┌──────────────┐
│ DeepSeek    │    │   Scoring     │    │  Seed Table  │
│ 按分类生成   │ →  │  Engine 评分  │ →  │  入库 (>=70) │
│ 25条候选     │    │  过滤 < 70   │    │  15-20条/类  │
└─────────────┘    └──────────────┘    └──────────────┘
```

### 2. Few-shot 注入策略: 相似度筛选
- **选择**: 从种子库中筛选与用户输入标题最相似的 3-5 条种子（基于 jieba 关键词重叠度），注入优化 prompt
- **理由**: 随机选择种子可能引入不相关风格，基于相似度筛选确保 few-shot 示例与用户意图匹配

### 3. 种子存储: PostgreSQL 新表 + Redis 缓存
- **选择**: `seed_titles` 表持久化存储，Redis 缓存活跃分类的种子列表（5 分钟 TTL）
- **理由**: 种子数据变更频率低（每周刷新），Redis 缓存可减少 DB 查询

```
seed_titles 表结构:
- id: UUID
- category: VARCHAR (游戏/生活/旅行/...)
- title: TEXT
- score: INTEGER (评分引擎打分)
- hook_type: VARCHAR (hook 类型标签)
- generated_at: TIMESTAMP
- is_active: BOOLEAN (软删除标记)
```

### 4. 刷新策略: 手动触发 + 定时任务
- **选择**: 提供 POST /api/v1/seeds/refresh 手动刷新端点，同时 Celery Beat 每周日凌晨 3 点自动刷新
- **理由**: 手动触发方便开发调试，定时任务保证种子库保持新鲜度

## Risks / Trade-offs

- **[种子标题同质化]** DeepSeek 可能生成风格相似的标题 → 生成 prompt 中加入多样性约束（要求不同 hook 类型、不同句式），评分后按 hook_type 分组，每组保留 top 2-3 条
- **[LLM 生成成本]** 每个分类生成 25 条，10 个分类 = 250 次 API 调用 → 每周一次，成本可控；限制每次生成使用 cheap 模式
- **[种子质量争议]** 评分引擎可能给不合适的标题打高分 → 预留人工审核接口（is_active 字段），后续可增加人工筛选环节
